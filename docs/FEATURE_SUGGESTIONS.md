# Mermaid Diagram Editor — Feature Suggestions

> Based on a thorough analysis of the existing codebase. Each suggestion is grounded in a real user problem.

---

## Tier 1 — High Value, Directly Solvable Problems

---

### 1. Diagram Version History

**Problem:** Users accidentally overwrite a working diagram and have no way back beyond OS-level undo.

**Solution:** Auto-snapshot on save (keep last N versions per file). A timeline panel shows past versions with diff highlighting (nodes/edges added/removed). One-click restore.

**Why it matters:** Diagrams evolve during meetings and workshops — people frequently want to go back to "what we had 10 minutes ago."

---

### 2. Code-to-Diagram Reverse Engineering

**Problem:** Developers waste time manually translating code structure into diagrams.

**Solution:** Paste or drag C# / TypeScript / Python source code → auto-generate a class diagram, dependency graph, or call graph. Similarly, paste an OpenAPI/Swagger YAML → generate a sequence diagram of API flows.

**Why it matters:** This is the #1 reason developers reach for a diagram tool. Existing architecture: the AI services already have a pipeline for generation — this adds a structured parsing path alongside the LLM path.

---

### 3. Custom Diagram Theme Editor

**Problem:** All diagrams look the same (default Mermaid.js theme), making it hard to match company branding or distinguish between diagram types.

**Solution:** A theme settings panel that exposes Mermaid.js CSS variables (node colors, fonts, border styles, line colors). Save named themes and assign per-diagram or globally. The existing `MarkdownStyleSettingsService` pattern can be extended.

**Why it matters:** Professionals sharing diagrams with clients or in presentations need them to look polished and on-brand.

---

### 4. Reusable Snippet / Component Library

**Problem:** Users repeatedly type the same boilerplate (e.g., standard swimlane structure, recurring node sets, company-standard subgraph layouts).

**Solution:** A sidebar panel where users can save named snippets. Single-click inserts the snippet at cursor. Snippets can be tagged by diagram type. The 8 built-in templates already exist — this extends that concept to user-owned snippets.

**Why it matters:** Power users working on related diagrams (e.g., a system architecture with recurring components) save significant time.

---

### 5. Share / Embed as URL

**Problem:** Sharing diagrams requires sending files, and recipients need the app or must open them online manually.

**Solution:** A "Share" button that encodes the diagram as a URL (same encoding as `mermaid.live`) and copies it to clipboard. Also generate an `<iframe>` embed snippet for wikis/docs. Optional: open in browser directly.

**Why it matters:** Teams working in Confluence, Notion, GitHub wikis, or sending diagrams in Slack need frictionless sharing without file attachments.

---

## Tier 2 — Significant Value, Moderate Complexity

---

### 6. AI: "Explain This Diagram" in Plain English

**Problem:** Diagrams received from others (or generated by AI) are hard to interpret quickly.

**Solution:** A button that sends the current Mermaid code to the AI and returns a plain-English explanation: what the diagram represents, what the key flows/relationships are, and any notable patterns. The AI service already supports custom prompts — this is a new prompt template.

**Why it matters:** Onboarding new team members to existing architecture diagrams is a known pain point. "What does this diagram mean?" is a very natural question.

---

### 7. AI: Convert Between Diagram Types

**Problem:** Users often realize mid-way that a flowchart would be better as a sequence diagram, or a class diagram would be clearer as an ER diagram.

**Solution:** "Convert to..." dropdown that uses AI to reinterpret the current diagram in a different diagram type, preserving as much semantic meaning as possible. The existing `DetermineDiagramTypeAsync` service is a natural anchor for this.

**Why it matters:** Diagram type decisions are made early and changing them manually is tedious and error-prone.

---

### 8. Multi-Diagram Project File

**Problem:** Complex systems require 5–10 related diagrams (context, container, component levels, sequence flows). These are currently managed as separate files with no relationship.

**Solution:** A `.mmdproj` project format that bundles multiple diagram files with metadata (diagram name, description, order). A project explorer panel replaces or augments the tab bar for project-scale work.

**Why it matters:** Architecture documentation (C4 model, for example) inherently requires a set of related diagrams. Users currently maintain a folder of files with no structure.

---

### 9. Interactive Diagram Mode (Clickable Nodes)

**Problem:** Static diagrams can't convey drill-down or navigation relationships — users have to mentally map "this node = that diagram."

**Solution:** In the preview, nodes can be annotated with links to other diagrams or external URLs. Clicking a node in presentation/fullscreen mode navigates to the linked diagram or opens the URL. Configuration via metadata comments in the Mermaid source (`%%link NodeA -> other_diagram.mmd`).

**Why it matters:** Architects presenting system overviews need to drill down into subsystems on demand during discussions.

---

### 10. Presentation Annotation Mode

**Problem:** During presentations, users want to highlight, circle, or draw attention to specific parts of a diagram in real time — without modifying the actual diagram.

**Solution:** An overlay drawing layer in presentation/fullscreen mode. Freehand draw, highlight, add arrows, type callouts. Annotations are ephemeral (cleared on exit) and never touch the diagram source.

**Why it matters:** This is a standard capability in tools like PowerPoint and Miro that users miss when presenting live diagrams.

---

## Tier 3 — Targeted but High-Impact for Specific Users

---

### 11. Diagram Complexity Metrics Panel

**Problem:** Large diagrams become unmaintainable and users have no signal for when a diagram is "too complex" to be useful.

**Solution:** A metrics sidebar showing: node count, edge count, subgraph depth, estimated render time, and a "complexity score." Flag diagrams that exceed readable thresholds with a suggestion to split.

**Why it matters:** Enforces diagramming best practices and helps teams maintain documentation quality over time.

---

### 12. Bulk / Batch Export

**Problem:** Users with a folder of `.mmd` files have to open and export each one individually.

**Solution:** "Batch Export" dialog: select a folder, choose output format (SVG / PNG / PDF), choose naming convention, and export all in one operation. Pairs well with the existing `ExportService`.

**Why it matters:** Documentation pipelines and CI/CD workflows that auto-generate diagram images from source need batch capability.

---

### 13. Diagram Diff Viewer

**Problem:** When reviewing changes to a diagram in code review (PR), text diffs of Mermaid code are hard to interpret visually.

**Solution:** A side-by-side diff view that renders both versions of a diagram and highlights what changed structurally (nodes added/removed/renamed, edges changed). Can be triggered by comparing clipboard content or two open tabs.

**Why it matters:** Engineering teams that version-control their diagrams in Git currently have no good way to review diagram changes in PRs.

---

### 14. Export to PDF

**Problem:** SVG and PNG are the only export options. Many enterprise workflows require PDF.

**Solution:** Add PDF export using the existing WebView2 print pipeline (`WebView2.CoreWebView2.PrintToPdfAsync`). This is low implementation cost given existing infrastructure.

**Why it matters:** PDFs are the standard for sharing architecture documents with non-technical stakeholders, clients, and compliance teams.

## Recommended Starting Points

The best features to implement first — **low effort, high daily impact**:

1. **#5 Share as URL** — Single button, URL encoding is straightforward, immediately useful for any team
2. **#14 PDF Export** — WebView2 already supports `PrintToPdfAsync`, minimal new code required
3. **#4 Snippet Library** — Extends the existing template system, clear UX pattern already in place
